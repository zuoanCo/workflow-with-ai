## 2. 代码组织目标

| 目录 | 职责 |
| --- | --- |
| `src/domain/` | 角色权限、工作流、成果、Provider、监控等领域类型与纯函数 |
| `src/data/` | MVP mock 数据、后续 API DTO 映射 |

## 3. 各切片验收口径

### 3.1 工程治理

- 明确 `main`、`feature/*`、`fix/*`、`release/*` 分支用途。
- 明确 PR 必须通过 `npm run lint`、`npm run build`、`npm run test`。
- 明确正式合入前必须至少 1 个 review approval。

### 3.2 账户与权限

- Owner、Producer、Reviewer、Viewer 的权限差异在 UI 与领域规则中可验证。
- 项目级授权和关键操作审计入口存在。
- 非授权操作有明确禁用或拒绝状态。

### 3.3 创作与 AI 模块

- 对话、生图、生视频共享统一任务状态模型。
- Provider 调用通过 Adapter 边界表达，页面不直接耦合第三方 SDK。
- 生成结果能进入资产或成果结构。

### 3.4 工作流

- 节点类型覆盖 BriefInput、PromptTemplate、ChatGenerate、ImageGenerate、VideoGenerate、ReviewGate、AssetTransform、PackageChannel。
- DAG 运行状态、失败原因、重试入口和人工审核卡点可展示。
- 节点输入输出结构可被单元测试覆盖。

### 3.5 成果、监控与打包

- 成果支持状态、版本、渠道、评分、返修记录。
- 监控覆盖成本、耗时、成功率、风险命中和队列状态。
- 打包适配器边界能区分 Web、公众号、短视频、小红书和下载包。

## 4. 验证策略

每个开发切片完成后运行最小验证：

```bash
npm run lint
npm run build
npm run test
```

若本地环境缺少依赖，先运行 `npm install`；若验证失败，必须先记录失败原因并修复当前切片，不得推进下一项。
