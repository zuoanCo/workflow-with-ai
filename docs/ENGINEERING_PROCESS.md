# Workflow AI 软件工程流程标准

## 执行原则

Workflow AI 项目必须严格执行五阶段流程：

```text
需求 → 设计 → 开发 → 测试 → 上线
```

任一阶段未通过检查点，不得进入下一阶段。当前仓库中的原型代码只能作为需求澄清和技术预研参考，不代表正式开发阶段已经开始。

## 阶段一：需求

负责人：常威PM  
目标：把 CEO 原始需求转化为可审批 PRD。

输出物：

- 原始需求记录。
- 需求分析报告。
- PRD v1.0。
- 内审结论。
- CEO 审批记录。

PRD 必须包含：

- 背景。
- 目标。
- 用户角色。
- 用户故事。
- 功能列表。
- 非功能需求。
- 优先级。
- 风险与约束。
- 验收标准。

检查点：

- PRD 未经 CEO 审批，不得进入设计。

## 阶段二：设计

负责人：农夫TA + 小笼包UI  
目标：完成技术方案和 UI 设计。

输出物：

- 技术架构文档。
- 数据模型。
- API 设计。
- AI Provider Adapter 设计。
- 工作流引擎设计。
- UI 设计稿。
- 设计自审与确认记录。

检查点：

- 技术方案未确认，不得开发后端与核心架构。
- UI 设计未确认，不得开发正式页面。

## 阶段三：开发

负责人：海飞丝FS  
目标：按设计文档实现代码。

输出物：

- 任务拆解。
- 代码实现。
- 单元测试。
- PR。
- Code Review 记录。

检查点：

- 所有正式代码必须来自已确认设计。
- PR 必须至少 1 个 review approval。
- 未通过构建、测试、lint 不得合入。

## 阶段四：测试

负责人：特仑苏TE  
目标：验证系统符合需求且无重大缺陷。

输出物：

- 测试用例。
- 测试记录。
- 缺陷列表。
- 回归测试记录。
- 测试报告。

检查点：

- P0/P1 未修复不得上线。
- 测试用例覆盖率目标不低于 90%。

## 阶段五：上线

负责人：海飞丝FS + 农夫TA  
目标：安全发布生产环境。

输出物：

- 上线计划。
- 回滚方案。
- 上线评审记录。
- 部署记录。
- 监控验证。
- 上线报告。

检查点：

- 必须 CEO 审批后上线。
- 错误率、响应时间、核心功能异常触发回滚。

## 上下文续跑约束

每次恢复任务必须先读取：

- `docs/ENGINEERING_PROCESS.md`
- `docs/TODO.md`
- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `docs/CONTINUITY.md`

续跑只能处理当前阶段的未完成事项。若当前阶段门禁未通过，禁止处理下一阶段任务。
