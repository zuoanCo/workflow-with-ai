## 1. 背景

AI 内容生产已经从单点生成进入流水线生产阶段。单独的对话、生图、生视频工具无法解决团队协作、权限控制、成本监控、成果审核、跨平台发布和长期复用问题。

Workflow AI 目标是参考 LibTV 一类连续内容生产玩法，把创作过程做成可编排、可审核、可监控、可复用的系统。系统优先交付 Web 工作台，同时为桌面端、移动端和 API Worker 预留扩展边界。

## 2. 产品目标

1. 建立统一 Web 工作台，覆盖 AI 对话、AI 生图、AI 生视频和成果管理。
2. 用画布工作流串联 Brief、脚本、素材、生成、审核、返修、发布打包。
3. 提供组织、账户、角色权限、项目隔离、密钥隔离和关键操作审计。
4. 提供成果资产管理能力，支持版本、状态、评分、渠道打包和交付记录。
5. 提供数据监控能力，覆盖成本、成功率、队列、审核风险和产出状态。
6. 保持高可维护性、高可测试性、高可扩展性和精美但高密度的后台式 UI。

## 3. 非目标

1. v1.0 不直接承诺完整自动发布到所有外部平台，优先完成打包与交付边界。
2. v1.0 不把模型密钥放入前端或本地明文配置。
3. v1.0 不以营销官网为核心形态，首屏应直接进入可工作的生产后台。
4. v1.0 不把当前原型代码视为正式开发交付，正式开发必须等待设计确认后启动。

## 4. 用户角色

| 角色 | 核心诉求 | 权限边界 |
| --- | --- | --- |
| Owner | 管理组织、成员、权限、密钥、成本和发布审批 | 最高权限，可审批发布和关键配置 |
| Producer | 创建项目、运行工作流、管理素材和成果 | 可编辑项目与运行任务，不可读取密钥明文 |
| Reviewer | 审核成果、标注风险、退回返修 | 可变更审核状态，不可修改组织级配置 |
| Viewer | 查看成果、数据和项目状态 | 只读权限 |

## 5. 用户故事

1. 作为 Owner，我需要管理组织成员、角色权限和平台密钥，确保创作系统在可控范围内运行。
2. 作为 Owner，我需要查看成本、成功率、风险命中和队列状态，及时发现异常消耗或生产阻塞。
3. 作为 Producer，我需要把内容 Brief 编排成 AI 对话、生图、生视频和审核节点，减少重复操作。
4. 作为 Producer，我需要复用提示词模板、参考素材和历史成果版本，提高连续内容生产效率。
5. 作为 Reviewer，我需要查看成果版本、风险标记和质量评分，并能通过、拒绝或退回返修。
6. 作为发布负责人，我需要把成果按 Web、公众号、短视频、小红书等渠道进行打包和交付。

## 6. 功能需求

| 模块 | 优先级 | 需求说明 | 验收口径 |
| --- | --- | --- | --- |
| 账户与组织 | P0 | 支持组织、成员、项目空间、成员状态和角色绑定 | Owner 可完成成员查看、角色分配和项目归属管理 |
| 权限体系 | P0 | 支持 RBAC、项目级授权、成果级审核权限和关键操作审计 | 不同角色只能访问授权功能，关键操作有审计记录 |
| AI 对话 | P0 | 支持模型路由、提示词模板、会话资产化和结果入库 | Producer 可创建会话并将结果沉淀为项目资产 |
| AI 生图 | P0 | 支持参考图、风格模板、批量生成、版本对比和审核流转 | 可生成多版本图片并进入成果审核状态 |
| AI 生视频 | P0 | 支持分镜、首尾帧、镜头提示词、渲染队列和失败重试 | 长任务有队列状态、失败原因和重试入口 |
| 画布工作流 | P0 | 支持 DAG 节点、人工审核节点、失败重试、成果归档 | 用户可在画布编排并运行完整创作流程 |
| 成果管理 | P0 | 支持成果状态、版本、渠道、评分、返修记录和下载交付 | Reviewer 可审核，Producer 可按渠道查看和导出 |
| 数据监控 | P1 | 支持成本、时延、成功率、风险命中率和队列状态 | Owner 可看到组织和项目维度监控指标 |
| 多平台打包 | P1 | Web 首发，预留 Desktop、Mobile、API Worker 和渠道适配器 | 同一成果可按目标渠道生成交付包 |

## 7. 核心流程

### 7.1 工作流生产流程

1. Producer 创建项目并录入 Brief。
2. Producer 在画布中选择提示词、对话、生图、生视频、审核、打包节点。
3. 系统按 DAG 执行节点，记录输入、输出、耗时、成本、状态和失败原因。
4. 生成结果进入成果管理模块，形成版本和审核记录。
5. Reviewer 审核成果，通过、拒绝或退回返修。
6. 通过审核的成果进入渠道打包和交付。

### 7.2 权限与审计流程

1. Owner 创建组织并邀请成员。
2. Owner 为成员分配角色和项目权限。
3. 系统对密钥读取、发布审批、删除成果、审核状态变更等操作写入审计。
4. 非授权角色访问受限能力时，系统拒绝并记录安全事件。

### 7.3 成果返修流程

1. Reviewer 对成果标注问题、风险和评分。
2. Producer 基于反馈创建新版本或重新运行部分节点。
3. 系统保留旧版本、返修原因、新版本产出和最终审核状态。

## 8. 非功能需求

| 类型 | 要求 |
| --- | --- |
| 可维护性 | 领域数据、工作流执行、AI Provider、UI 展示分层；避免业务逻辑散落在页面组件内 |
| 可测试性 | TypeScript 类型检查、核心领域逻辑单元测试、关键流程后续补充 E2E |
| 可扩展性 | AI Provider Adapter、工作流节点插件化、渠道发布适配器、打包适配器 |
| 安全性 | 密钥只保存在服务端；前端只拿短期任务令牌；权限最小化；关键操作审计 |
| 稳定性 | 长任务支持队列、幂等、重试、失败补偿和断点续跑 |
| 体验质量 | 高密度后台式信息布局，避免空泛营销页；核心任务入口清晰，状态反馈明确 |
| 可观测性 | 记录任务成本、耗时、成功率、队列状态、错误原因和风险命中 |

## 9. 优先级

### P0

- 账户与组织。
- 权限体系。
- AI 对话。
- AI 生图。
- AI 生视频。
- 画布工作流。
- 成果管理。

### P1

- 数据监控。
- 多平台打包。
- 渠道适配器扩展。
- 桌面端、移动端和 API Worker 打包边界。

### P2

- 高级运营分析。
- 自动化发布到更多外部平台。
- 组织级模板市场。
- 更细粒度的成本预测与预算控制。

## 10. 验收标准

### 阶段一：需求验收

- PRD 包含背景、目标、用户角色、用户故事、功能列表、非功能需求、优先级、风险与约束、验收标准。
- PRD 经过内审，内审问题已记录并处理。
- CEO 明确审批通过后，才允许进入设计阶段。

### 阶段二：设计验收

- 技术架构、数据模型、API、AI Provider Adapter、工作流引擎、多端打包方案完成并确认。
- UI 信息架构和关键页面设计稿完成并确认。

### 阶段三：开发验收

- 正式代码实现符合已确认设计。
- 构建、测试、lint 通过。
- PR 获得至少 1 个 review approval。

### 阶段四：测试验收

- 测试用例覆盖率目标不低于 90%。
- P0/P1 缺陷清零。
- 回归测试通过并形成测试报告。

### 阶段五：上线验收

- 上线计划、回滚方案、监控验证齐备。
- CEO 审批通过后上线。
- 上线后核心功能、错误率、响应时间和队列状态验证通过。

## 11. 风险与约束

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 模型能力和成本不稳定 | 产出质量、预算和交付周期不可控 | 通过 Provider Adapter、成本监控、预算阈值和降级策略兜底 |
| 生视频任务耗时长 | 队列积压、失败重跑成本高 | 引入队列、重试、断点续跑、失败补偿和任务状态可视化 |
| 多平台规则变化快 | 打包或发布逻辑频繁变更 | 用渠道适配器隔离平台规则 |
| 密钥与素材敏感 | 泄露会造成安全与商业风险 | 密钥服务端保存、权限最小化、关键操作审计 |
| 当前原型先于流程存在 | 容易绕过需求和设计门禁 | 明确原型仅作预研参考，正式开发必须等待 PRD 和设计审批 |
| UI 过度营销化 | 不适合高频生产工作台 | 采用后台式信息架构和高密度任务流设计 |
