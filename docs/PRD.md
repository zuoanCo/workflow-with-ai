# Workflow AI PRD v0.1（需求阶段草案）

状态：草案，待内审，待 CEO 审批。  
流程约束：本 PRD 未经 CEO 审批前，不得进入设计或正式开发。

## 背景

AI 内容生产已经从单点生成进入流水线生产阶段。单独的对话、生图、生视频工具无法解决团队协作、权限控制、成本监控、成果审核、跨平台发布和长期复用问题。

Workflow AI 目标是参考 LibTV 一类连续内容生产玩法，把创作过程做成可编排、可审核、可监控、可复用的系统。

## 目标

1. 提供统一 Web 工作台，覆盖 AI 对话、AI 生图、AI 生视频。
2. 用画布工作流把 Brief、脚本、素材、生成、审核、发布串联起来。
3. 提供账户管理、角色权限、密钥隔离和审计能力。
4. 管理成果资产，支持版本、状态、评分、渠道打包。
5. 为 Web、桌面、移动端和 API Worker 预留清晰扩展边界。

## 用户角色

- Owner：管理组织、成员、权限、密钥、计费和发布审批。
- Producer：创建项目、运行工作流、管理素材和成果。
- Reviewer：审核内容、标注风险、退回返修。
- Viewer：查看成果和数据，不具备写权限。

## 用户故事

1. 作为 Owner，我需要管理组织成员、角色权限和平台密钥，确保创作系统在可控范围内运行。
2. 作为 Producer，我需要把一个内容 Brief 编排成 AI 对话、生图、生视频和审核节点，减少重复操作。
3. 作为 Reviewer，我需要查看成果版本、风险标记和质量评分，并能退回返修。
4. 作为运营负责人，我需要查看成本、成功率、队列、审核风险和各渠道成果状态。
5. 作为发布负责人，我需要把成果按 Web、公众号、短视频、小红书等渠道进行打包和交付。

## 核心功能

| 模块 | 优先级 | 说明 |
| --- | --- | --- |
| 账户与组织 | P0 | 组织、成员、项目空间、角色绑定。 |
| 权限体系 | P0 | RBAC + 项目级授权 + 成果级审核权限。 |
| AI 对话 | P0 | 模型路由、提示词模板、会话资产化。 |
| AI 生图 | P0 | 参考图、风格模板、批量生成、版本对比。 |
| AI 生视频 | P0 | 分镜、首尾帧、镜头提示词、渲染队列。 |
| 画布工作流 | P0 | DAG 节点、人工审核、失败重试、成果归档。 |
| 成果管理 | P0 | 成果状态、版本、渠道、评分、返修记录。 |
| 数据监控 | P1 | 成本、时延、成功率、风险命中率、队列状态。 |
| 多平台打包 | P1 | Web 首发，预留 Desktop、Mobile、API Worker。 |

## 非功能需求

- 可维护性：领域数据和 UI 分层，后续可迁移到后端 API。
- 可测试性：TypeScript 类型检查、组件测试、后续增加 E2E。
- 可扩展性：AI Provider Adapter、工作流节点插件化、渠道发布适配器。
- 安全性：密钥不进前端包，权限最小化，关键操作审计。
- 体验质量：后台式高密度信息布局，避免空泛营销页。

## 验收标准

- 阶段一验收：PRD 内容完整，风险明确，CEO 审批通过。
- 阶段二验收：技术方案和 UI 设计经过确认。
- 阶段三验收：代码实现符合设计，构建、测试、lint 通过，PR 获得 review approval。
- 阶段四验收：测试报告通过，P0/P1 清零。
- 阶段五验收：上线计划、回滚方案、监控验证齐备，CEO 批准上线。

## 风险与约束

- 模型能力和成本不稳定，需要 Provider Adapter 与成本监控兜底。
- 生视频任务耗时长，需要队列、重试、断点续跑和失败补偿。
- 多平台发布规则变化快，需要渠道适配器隔离。
- 密钥和素材涉及敏感资产，必须服务端保存并记录审计。
- 当前仓库已有代码仅作为原型，不代表正式开发阶段开始。
